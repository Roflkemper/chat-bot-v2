# Протокол ведения SESSION_LOG

## Зачем

`docs/SESSION_LOG.md` — это **human context**: что и почему было сделано в этой сессии. Восстанавливается по нему память между сессиями: новый чат, переезд на новую машину, ввод нового сотрудника.

Авто-генерируемый `docs/CHANGELOG_DAILY.md` его **не заменяет** — там только цифры (PnL, restarts, pipeline funnel), без объяснения «почему отключил X», «какой инсайт получил», «к чему пришёл».

## Когда писать

Обязательно — **в конце любой сессии**, где было сделано что-то одно из:

- Изменена логика (новая стратегия, новый фильтр, отключение детектора)
- Изменён конфиг (.env.local, threshold'ы, флаги)
- Найден инсайт по результатам (бэктест, paper-trade анализ, наблюдение из live-данных)
- Закрыт инцидент или открыт новый
- Большой рефакторинг или migration
- Запущен крупный эксперимент (E1 GA, E2 LLM-narrator и т.п.)

Не писать — для:
- Косметические правки документов без изменения логики
- Авто-обновления (CHANGELOG_DAILY и т.п.)
- Чисто инфраструктурные правки без эффекта на торговлю

## Как писать

Через helper-скрипт:

```bash
python scripts/append_session.py "TZ-XXX: краткое описание" "Тело: что сделано, почему, какие риски"
```

Или интерактивно:

```bash
python scripts/append_session.py
```

Скрипт добавит запись с **UTC timestamp** в `docs/SESSION_LOG.md`.

## Формат записи

```markdown
## YYYY-MM-DD (HH:MM UTC) — TZ-XXX: заголовок

**Контекст:** что мотивировало (наблюдение, инцидент, баг).

**Сделано:**
- пункт 1
- пункт 2

**Решения / уроки:** что узнали, что изменили в подходе.

**Открытые вопросы:** что отложено, какие риски остались.
```

Минимум — заголовок + 2-3 строки тела. Максимум — на разумном пределе (если запись на полстраницы — это норм).
