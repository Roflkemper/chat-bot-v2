## 3a. Root cause analysis — WHY RED (v2)

**The v2 RED verdict is primarily a OHLCV resolution artifact, not an engine logic bug.**

### Primary cause: 1-minute bar resolution too coarse for 0.03% grid step

The engine uses `grid_step_pct / 100.0` internally, so:
- `grid_step_pct = 0.03` → step = 0.0003 = 0.03% ≈ 22 USDT per level at BTC ~74,155

BTC typical 1-minute ATR ≈ 100–200 USDT in active markets.
- In 1 minute, BTC may cross 4–9 grid levels (100–200 / 22 = 4.5–9 oscillations)
- The real bot executes on **tick data** — captures each level crossing immediately
- The 1-minute sim captures only the net end-of-bar position → misses ~75% of crossings

Evidence:
- Real/sim trade ratio: TEST_1=4.1x, TEST_2=12x, TEST_3=1.9x, SHORT_1.1%=5.3x
- Average real rate: TEST_1 = 1428 OUT / (13d × 24h) = 4.6/h
- Average sim rate: TEST_1 = 347 / (13d × 24h) = 1.1/h
- Expected ratio from tick vs 1m: (100 USDT ATR) / (22 USDT step) ≈ 4.5× ✓ matches observed

### Secondary cause: PnL sign inversion in sim

Sim generates net negative PnL while real bots are profitable. This is a consequence of:
1. Under-trading: real bot completes far more IN-OUT profitable cycles per day
2. Same stop losses fire in sim at lower trade frequency → stop-loss PnL drag dominates
3. Extended drawdown periods in April 2–16 (pre-snapshot) where sim holds losing positions
   that the real bot may have handled differently via operator intervention

### Why engine logic is likely OK

- Previous W1/W2/W3 validation on April 17 & 20 with single windows passed
- The engine grid mechanics (grid_step_pct/100, target, instop) match GINAREA_MECHANICS docs
- The under-trading ratio (4–12x) matches the expected tick-vs-1m oscillation count

### Required fix for meaningful reconciliation

**TZ-ENGINE-FIX-RESOLUTION**: Use 1-second OHLCV (or tick-level) data for reconcile with 0.03% grid bots.
The 1-minute bar resolution is fundamentally too coarse relative to the grid step size.
Alternatively: validate only on windows where BTC was range-bound (ATR < 5× grid step).

Priority:
1. **TZ-ENGINE-FIX-RESOLUTION** — primary: 1s bars for grid bots with step < 0.1%
2. **TZ-ENGINE-FIX-INSTOP-SEMANTICS-B** — secondary: verify LONG bots instop mode
