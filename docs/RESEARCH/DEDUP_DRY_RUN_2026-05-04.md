## §1 What this dry-run is and isn't

**What it IS:** the closest-available production log to the kind of stream the dedup wrapper was designed for. The `decision_log/events.jsonl` is generated by `services/decision_log/event_detector.py` and represents a real, time-stamped, structured stream of bot/portfolio events, many of which feed downstream alert emitters.

**What it ISN'T:** the *direct* output of the auto_edge_alerts / setup_detector / protection_alerts emitters that TELEGRAM_EMITTERS_INVENTORY.md flagged for DEDUP-WRAP. Those emitters don't write structured JSONL — they call `bot.send_message()` directly. So this dry-run is a **structural proxy**: the same wrapper logic, applied to comparable event types.

**Implication:** suppression rates here estimate "how aggressively the wrapper would intervene if the same configs were applied to the comparable emitter classes." Absolute numbers are **not** a 1:1 forecast of how many auto_edge_alerts would be suppressed, but the *relative ordering* across event types and the *pattern* (state-suppress vs cooldown-suppress) carry over.

**Pattern:** in every event type that suppresses meaningfully, **state-unchanged dominates over cooldown**. This is exactly Finding 1 from the inventory: when the underlying condition isn't changing materially, the cooldown alone (which is what the codebase already has on protection_alerts and auto_edge_alerts) lets repeats through; the state-change layer does the heavy lifting.

For BOUNDARY_BREACH, 94% of suppressions are state-unchanged (926 of 989). Translation: same price level keeps re-breaching by tiny amounts; without state-change check those would all reach the operator.

### 4.2 PNL_EVENT (sample from 724 suppressed)

PNL_EVENTs cluster when the portfolio sits in a slow drift — `delta_pnl_usd` changes by tens of dollars per 15-min check, well below the $200 threshold. The first emit fires; subsequent emits are suppressed until either (a) the delta swings by ≥$200 from the last emitted value (state-change), or (b) a fresh swing happens after cooldown expiry.

In this 4-day window, ~88% of PNL_EVENTs occurred during slow-drift periods. The 93 that **would** have emitted correspond to genuine PnL inflection points worth operator attention.

### 4.3 PNL_EXTREME (sample from 118 suppressed)

Even more extreme suppression (99.2%) because PNL_EXTREME with $500 threshold + 30-min cooldown + state-change is highly selective by design. In this 4-day window, only **1 of 119 events** would have reached the operator post-dedup. Combined with the other types, this matches the "tons of detail" complaint: PNL_EXTREME was contributing very little signal but lots of noise.

## §5 Edge case: POSITION_CHANGE behaving healthily

POSITION_CHANGE (188 events, 10.6% suppress) is the **only** event type in the healthy 10–70% band. This is structurally interesting:

- POSITION_CHANGE fires when bot positions shift (new fills, partial closes). Each event represents an actual trade, not a re-evaluation of an unchanging condition.
- The state-change check (Δ ≥ 0.05 BTC) suppresses only the "small-fill clusters" — when 5 bots close 0.001 BTC each within 5 minutes.
- Most POSITION_CHANGEs are real, distinct events → low suppression rate is the correct outcome.

This is the comparison point: **the wrapper isn't broken for POSITION_CHANGE**. Where it's "too aggressive" elsewhere, the source pattern is *the data is genuinely repetitive*, and suppression is the right response.

## §6 Recommendations (D71)

| Event type | Default config tuning | Wire-to-prod readiness |
|------------|----------------------|------------------------|
| PNL_EVENT | **Loosen state Δ to $400-500** (current $200 too tight; 88% suppress means even moderate PnL drifts get masked). With $500: estimate ~60% suppress (operator validates). | Wire after Δ-tuning |
| BOUNDARY_BREACH | **Either cluster-collapse** (multiple bots at same price → one alert) **or loosen state Δ to $200-300**. 95% suppress means current dedup is over-eager — but the underlying flap is real, so cluster collapse is the correct fix. | Wire WITH cluster-enabled, current Δ probably fine |
| POSITION_CHANGE | **Keep as-is** — 10.6% in healthy band. | ✅ Ready for wire |
| PNL_EXTREME | **Drop entirely OR loosen massively.** 99.2% suppress means current rule is essentially "never emit." Either operator wants 0-1 PNL_EXTREME alerts/4-days (current behavior) or these events shouldn't be on the dedup path at all. | Operator decides scope first |
| PARAM_CHANGE | **No dedup needed** — 0% suppress, 68 events in 4 days = 17/day = manageable raw. | Drop wrapper for this type |
| BOT_STATE_CHANGE | **No dedup needed** — 17 events in 4 days. | Drop wrapper |
| REGIME_CHANGE | **No dedup needed** — 5 events in 4 days; these are exactly the events you want every time. | Drop wrapper |

**Summary direction:** dedup wrapper is **not one-size-fits-all**. Three tiers emerge:

1. **High-pressure types (PNL_EVENT, BOUNDARY_BREACH, PNL_EXTREME):** wrap, but loosen current default thresholds — 88-99% suppression at default is too aggressive. Cluster collapse (already implemented) should be enabled for BOUNDARY_BREACH.
2. **Healthy types (POSITION_CHANGE):** wrap as-is.
3. **Low-frequency types (PARAM_CHANGE, BOT_STATE_CHANGE, REGIME_CHANGE):** **don't wrap.** The wrapper costs more than it returns when the input rate is already 1-17/day.

## §7 Caveats

1. **Decision-log events ≠ Telegram alerts.** This dry-run shows what the wrapper would do to the structured event stream. Real auto_edge_alerts / setup_detector emitters fire on different (price-tick / RSI-tick) cadence; their suppression rates will differ.
2. **4-day window** captures recent operator activity but isn't a calendar-month sample. Slow-news days and macro-event days produce very different event mixes. Re-run on a week or month for stability check.
3. **Default thresholds were guessed**, not calibrated. The brief specified the wrapper logic; per-type thresholds in this dry-run are operator-overridable. Tuning them is the next step.
4. **No MARKDOWN-heavy day in window.** The 4 days were all relatively bullish/range. Bear days may produce different suppression profiles (more BOUNDARY_BREACH at lower bound, etc.).
5. **Suppression of `bot_id="multiple"`.** All 1,041 BOUNDARY_BREACH events have `bot_id="multiple"`, so they all share key=`global` in the wrapper. A cluster-aware approach that splits by individual bot_id from the payload's `affected_bots` list would suppress less aggressively. Worth implementing before wire-up.
6. **Rough order-of-magnitude noise reduction:** if all wraps were applied with current defaults, operator Telegram volume would drop from 2,255 → 404 messages over 4 days = **−82% volume**. With recommended tuning (loosen PNL/BOUNDARY thresholds, drop wrap on low-freq types), expect ~−50-60% — still substantial but closer to the operator's "5-8 actionable per day" target from the brief.
