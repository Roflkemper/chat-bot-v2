# Глоссарий: технические коды → человеческие имена

Чтобы в TG-сообщениях, отчётах и обсуждениях все разговаривали на одном
языке, фиксируем русские названия для технических кодов стратегий и сценариев.

---

## Стратегии grid-ботов

| Технический код | Русское имя | Описание |
|---|---|---|
| `baseline` | **«Без вмешательства»** | Сетка работает как сейчас — никто не пауcтит, не выгружает, границы не двигает. Эталон для сравнения. |
| `pause_on_drawdown` | **«Защита от слива»** | Пауза новых ордеров при unrealized < -$30 + возврат после отката >50%. Жертвует объёмом ради защиты. |
| `partial_unload_on_retrace` | **«Частичная фиксация»** | Если был пик прибыли > $5 и она откатилась >40% — выгружаем 30% позиции. |
| `trend_chase` | **«Подъём границ за трендом»** | При движении цены >0.5% за час — поднимаем верхнюю границу сетки на 2%. Не даёт боту замёрзнуть за boundaries. |
| `combined` | **«Защита+Фиксация+Догон»** (комбинированная) | Все три правила вместе — лучший компромисс между объёмом и защитой. |

**В TG-сообщениях** используем русские имена. В коде — английские константы.

## Каскадные сетки и каскад-сетапы

⚠️ В проекте есть **4 разных «каскада»** — путать их нельзя. Каждый получает уникальное русское имя.

| Технический код / файл | Русское имя | Что это |
|---|---|---|
| `_backtest_cascade_grid.py` / `CASCADE_GRID_BACKTEST.md` | **«Лимитный каскад v1»** | 3-tier laddered лимитки (не GinArea-боты). Триггер по 30-min range. Tier1/2/3 = 1%/2%/3% от ref, фиксированные TP+SL. **PF 0.77, отвергнут.** |
| `_backtest_cascade_grid_v2.py` / `CASCADE_GRID_V2_BACKTEST.md` | **«Лимитный каскад v2 ladder-по-цене»** | Tier-2 открывается ТОЛЬКО когда цена ушла +2% от точки начала окна, Tier-3 — +3%. У каждого свой entry/TP/SL. **PF 1.15 (wide+trail), на грани.** |
| `_backtest_cascade_ginarea_v3.py` / `CASCADE_GINAREA_V3_BACKTEST.md` | **«Каскад сеток GinArea v3»** | 3 настоящих GinArea-бота с разными `target_profit_pct` (TD): Бот-1 быстрый (TD 0.2-0.25), Бот-2 средний (0.3-0.4), Бот-3 редкий (0.5-0.6). Общие: gs=0.03, period=30, thresh=0.3. **Лидер SHORT TD=(0.2, 0.35, 0.6) → +7 455.** |
| `_backtest_cascade_ginarea_v4.py` / `CASCADE_GINAREA_V4_BACKTEST.md` | **«Каскад сеток GinArea v4 (расширенный)»** | Расширенный sweep V3: 180 комбинаций gs × period × thresh × TD-триплет. **Лидер: gs=0.02, period=15, thresh=0.2, TD=(0.3, 0.5, 0.8) → +25 386, ROI/DD 4.42.** |
| `cascade_trade.py` / `CASCADE_PARAMS` | **«Сетап после каскада ликвидаций»** | Не сетка! Одиночная сделка-отскок после live LIQ_CASCADE: LONG buy с TP +1.14% / SL -0.5% / hold 12h при каскаде LONG-ликвидаций ≥5 BTC за 5 мин. Backtest n=103, 73% точность. **Интегрирован в paper_trader.** |
| `pre_cascade_alert` / `PRE_CASCADE_BACKTEST.md` | **«Алерт перед каскадом»** | Сигнал на основе funding + OI + LS до того как каскад начался. Precision 0% на N=3, **не используется в проде**. |

### Параметры терминологии (чтобы не путать)

| Параметр на платформе GinArea | В коде / бэктестах | Что значит |
|---|---|---|
| Шаг сетки (%) | `grid_step_pct`, `gs` | Расстояние между IN-ордерами в % от цены |
| Целевой уровень прибыли (%) | `target_profit_pct`, **`TD`** | Профит на ОДНУ грид-сделку в % |
| Инстоп (%) | `instop_pct` | Задержка открытия IN после отката от экстремума (см. GINAREA_MECHANICS.md §4) |
| Тейк-Профит (%) | TP / take_profit | Закрытие ВСЕЙ позиции на уровне в % от средней цены |
| Обновление Тейк-Профита (%) | take_profit_update | Автоинкремент TP после срабатывания |
| Indicator Period | `indicator_period` | Сколько 1-min баров смотреть для Price% |
| Indicator Threshold (%) | `indicator_threshold_pct`, **`thresh`** | Порог Price% для входа в цикл (start trigger) |

### Каскад GinArea v4 — конкретные параметры лидера (для оператора)

3 SHORT-бота на BTCUSDT параллельно:
- Все три: gs=0.02, indicator Price% TF=1m Period=15 > 0.2%, instop=0.03, min/max stop=0.01/0.04, order_size=0.003, order_count=800
- **Целевой уровень прибыли (TD):** Бот-1 = 0.30, Бот-2 = 0.50, Бот-3 = 0.80
- TP / Обновление: **off / off** (V4 их не варьировал — открытый вопрос для нового sweep'а)

## Применение в коде

**Не переименовываем** код-уровневые константы (это сломает API и тесты).
**Переименовываем** только то что видит оператор:
- TG-сообщения от сервисов
- Markdown отчёты в `docs/`
- Названия в TG-командах (`/bots_kpi`, `/audit`)

Для каждого технического кода в TG-форматтерах вызываем функцию-переводчик
`humanize_setup_type()` / `humanize_scenario()` (которые возвращают русское имя).
