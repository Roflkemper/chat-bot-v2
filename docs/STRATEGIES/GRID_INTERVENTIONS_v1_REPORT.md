**Скрипт:** `tools/_validate_engine_short_objem.py`

**Метод:** взяли точный live-конфиг ШОРТ ОБЬЕМ из `ginarea_live/params.csv`,

**Результат:**

| Метрика | Engine | Real | Δ% |

| volume_usd | $30,383 | $26,667 | +14% |

оптимизируем стратегию по **объёму, drawdown и position**, а не по PnL.

### 3. Бэктест 5 сценариев интервенций × 11 окон по 7 дней
**Скрипт:** `tools/_backtest_grid_interventions.py`

**Конфиг:** ШОРТ ОБЬЕМ live (target=0.13%, grid=0.02%, instop=0.02%, без границ).

**Результаты (среднее за окно 7 дней):**

| Сценарий | Avg Volume | Avg Net PnL | Worst DD | Avg Interventions |
|---|---:|---:|---:|---:|
| `baseline` (без интервенций) | $728k | -$905 | -$110 | 0 |
| `pause_on_drawdown` | $237k | **-$183** | -$110 | 194 |
| `partial_unload_on_retrace` | $688k | -$721 | -$110 | 980 |
| `trend_chase` | $642k | -$802 | -$110 | 494 |
| `combined` (все три) | **$400k** | **-$248** | -$110 | 1197 |

**Главные открытия:**

1. **`pause_on_drawdown` снижает убытки в 4.9× раз** ($-905 → $-183) ценой 67% объёма
2. **`combined` — лучший компромисс:** объём $400k (55% от baseline), убыток $-248 (3.6× меньше)
3. **На самых волатильных окнах разница огромная** (окно 5: baseline DD $-2811 vs pause DD $-462 = 6× меньше)
4. **`trend_chase` с RaiseBoundaryOnConfirmedTrend** не помогает у ШОРТ ОБЬЕМ — бот
   работает без границ изначально, правило не находит что менять
5. **На спокойных окнах** (окно 6, 9) интервенции не отличаются от baseline — правильно,
   они срабатывают только когда нужно

### 4. Live grid_coordinator с индикатором истощения движения
**Сервис:** `services/grid_coordinator/`
**Wired:** `app_runner.py`, запускается как async task каждые 5 минут

**Алгоритм:** проверяет 5 независимых сигналов истощения восходящего/нисходящего
движения:

1. **RSI(14) 1h** ≥ 75 + RSI начал снижаться (для верха)
2. **MFI(14) 1h** ≥ 75 на свежем баре
3. **Volume z-score** < 0 при новом 24h хайе (no volume confirmation)
4. **OI 1h** ≥ +1.0% AND **funding 8h** ≥ 0.04% (longs over-crowded)
5. **BTC↔ETH corr** ≥ 0.7 + **ETH RSI** ≥ 70 (синхронный пик)

Если **3 из 5 сигналов одновременно** → отправка TG-карточки:

```
🔝 ВЕРХ ИСТОЩАЕТСЯ (3/5 сигналов)

BTC: $80,500  RSI=78  MFI=76  vol_z=-0.5
OI 1h: +1.20%  funding 8h: +0.0500%
ETH RSI: 72  BTC↔ETH corr: 0.85

Сигналы: rsi_high_falling, mfi_high, eth_sync_high

Рассмотри закрытие SHORT-сеток сейчас и переоткрытие на откате.
Это рекомендация — бот не торгует, решение оператора.
```

**Cooldown:** 30 мин между алертами одного направления (нет спама).
**Bot не торгует** — только наблюдение и сигнал. Управление через GinArea API
будет следующим шагом когда оператор включит API-доступ.

**Журнал срабатываний:** `state/grid_coordinator_fires.jsonl`

**Тесты:** `tests/test_grid_coordinator.py` — 13/13 зелёных, включая live-evaluation
на BTC/ETH прямо сейчас.

**Live на момент написания:**
- BTC RSI 66.6, ETH RSI 64.1 — близко к overbought но ещё не там
- vol_z -2.11 (объём падает — раннее предупреждение)
- BTC↔ETH corr 0.816 (синхронные)
- upside_score 0/5 (норма), downside_score 0/5 (норма)

Если BTC дойдёт до RSI 75+ с новым хайем — coordinator пошлёт алерт.

## Что **не** сделано и почему

1. **Не оптимизировал PnL** — движок недостаточно точен на коротких окнах
   (расхождение ±$5 на $30k volume). Это не дефект движка — комиссии BitMEX
   зависят от maker/taker mix, который варьируется. На длинных окнах усреднится.
2. **Не предложил конкретные параметры миграции TEST_3 → новый бот** —
   нужны настоящие параметры рабочих ботов (есть в params.csv) и их валидация
   на твоей реальной торговле, не на одном эпизоде ШОРТ ОБЬЕМ.
3. **Не подключил GinArea API для управления** — у меня нет учёток в .env,
   и операция через API нужна твоего разрешения. Архитектура готова: добавить
   `client.start_bot()` / `client.stop_bot()` в grid_coordinator.
