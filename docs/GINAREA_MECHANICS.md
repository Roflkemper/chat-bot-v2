## 1. Параметры бота

### Режим значений

| Режим | Описание |
|---|---|
| % режим (default) | Параметры в % от текущей цены, адаптируются при движении. [UI docs] |
| Абсолютный | Параметры в абсолютных значениях, не пересчитываются. [UI docs] |

### Типы контрактов в портфеле

Алексей использует **два разных типа контрактов** для разных направлений: [user]

| Направление | Контракт | Qty в | PnL в | Пример бота |
|---|---|---|---|---|
| SHORT | **Linear USDT-M** (BTCUSDT) | BTC | USDT | TEST_1/2/3: 0.001 BTC |
| LONG | **Inverse coin-margined** (XBTUSD) | USD (контракты) | BTC | BTC-LONG-C: 100 USD |

**Почему такая асимметрия (рациональное объяснение):**
- Linear SHORT — PnL линеен в USDT, убыток не усиливается при росте BTC (inverse SHORT усиливал бы).
- Inverse LONG — PnL в BTC, при росте цены double-win (profit растёт в BTC + BTC дорожает в USD).

**Для движка:** PnL и liq-формулы разные для двух сторон. См. §7.

### Параметры сетки

| Параметр | Единица | Описание | Источник |
|---|---|---|---|
| `side` | enum | 1=LONG, 2=SHORT | [API] |
| `order_size` | BTC (linear) / USD (inverse) | Размер одного IN-ордера. Единица зависит от типа контракта | [UI docs] |
| `order_count` | шт | Макс. количество одновременно открытых IN | [UI docs] |
| `grid_step_pct` | % | Шаг сетки **от последнего открытого IN-ордера**. В % режиме — % от цены последнего IN. | [UI docs] |
| `multiplier` | коэфф ≥1 | Множитель шага. Макс. шаг ≤ 50% цены | [UI docs] |
| `order_multiplier` | коэфф ≥1 | Множитель размера. Сброс при закрытии всей позиции | [UI docs] |

### Параметры входа/выхода

| Параметр | Описание | Источник |
|---|---|---|
| `target_profit_pct` | Расстояние от цены открытия IN до триггера. При достижении — ставится stop-profit на `min_stop` | [UI docs] |
| `min_stop_pct` | Расстояние от триггера до stop-profit. `0` = стоп прямо на триггере | [UI docs] |
| `max_stop_pct` | Макс. отклонение комбо-trailing-stop от цены. `0` = trailing отключён | [UI docs] |
| `instop_pct` | **См. §4 — две семантики в зависимости от режима** | [UI docs] |

**Formula profit/IN (в %):** `target_profit_pct − min_stop_pct` — столько % приносит одна грид-сделка. [UI docs]

### Границы и защиты

| Параметр | Описание | Источник |
|---|---|---|
| `boundaries.lower/upper` | Диапазон создания новых IN. Существующие IN могут закрываться за пределами | [UI docs] |
| `dsblin` | При выходе за boundaries прекращает новые IN. После возврата НЕ возобновляет автоматически | [UI docs] |
| `tsl` | Trailing Stop Loss. Варианты: default / % of profit | [UI docs] |
| `take_profit` + `take_profit_update` | Закрытие всей позиции на уровне. `update` — автоинкремент после срабатывания | [UI docs] |

### Indicator (start trigger)

[UI docs] Поддерживает: EMA, RSI, MA, SMMA, ATR, ATR%, Supertrend, Price%.  
Комбинации: None / AND / OR. Exit indicator — отдельный набор, опционально.

**Режим проверки:**
- **«Разовая проверка»** (тумблер ON/OFF) — если ON, indicator проверяется **один раз на цикл**. Цикл начинается при старте бота или после **full-close** позиции. После full-close флаг «пройдена» сбрасывается, ждём новый indicator signal. [user]
- Out Stop ≠ full-close. Если после Out Stop остались открытые IN-ордера — бот продолжает работать без перепроверки indicator. Если Out Stop закрыл всю позицию — это full-close, цикл завершается.

## 2. Жизненный цикл IN-ордера

```
[Indicator trigger срабатывает (см. §5)]
      ↓
[Цена идёт против направления бота]
      ↓
[Instop check (grid): ждём разворот на instop_pct от экстремума] ← см. §4
      ↓
[IN-ордер открыт по рыночной цене entry_price]
      ↓
[Следующий уровень grid = последний IN ± grid_step_pct]
      ↓
[Цена идёт в прибыль → достигает entry_price ± target_profit_pct]
      ↓
[Выставляется stop-profit на расстоянии min_stop_pct от триггера]
      ↓
  ┌───────────────────────────────────────────────────┐
  │ Откат ≥ min_stop_pct                             │
  │ → IN закрыт. Profit = target − min_stop         │
  └───────────────────────────────────────────────────┘
  ┌───────────────────────────────────────────────────┐
  │ Движение без отката                               │
  │ → Новые IN открываются                           │
  │ → Стопы объединяются в Out Stop (§3)             │
  │ → Trailing с max_stop_pct                        │
  └───────────────────────────────────────────────────┘
```

## 3. Out Stop — объединение ордеров

[UI docs]

**Когда срабатывает объединение:** IN-ордера, достигшие своего триггера (target), комбинируются в общий Out Stop.

**Поведение при закрытии Out Stop:**
- Последние открытые IN могут закрыться в **минус** (цена не прошла их target).
- Ранее открытые IN дают **плюс**.
- Суммарно / взвешенно — позиция в плюсе. Средневзвешенный профит ≈ `target_profit_pct × N_orders` (за вычетом последних).

**Trailing max_stop_pct:**
1. После объединения стопов комбо-стоп trailing'ует за ценой.
2. Макс. отклонение комбо-стопа от цены = `max_stop_pct`.
3. При delta(цена − стоп) = `max_stop_pct` — стоп начинает следовать за ценой.
4. Каждый следующий IN, входящий в группу, закрывается с возрастающим профитом.

**Если `max_stop_pct = 0`:** trailing отключён, stop на триггере → фиксация на первом касании.

**Out Stop vs full-close:**
- Out Stop = закрытие **группы** IN-ордеров (тех, что достигли target и объединились).
- Если после Out Stop в позиции остались IN, которые target не достигли — это **не** full-close.
- Full-close = закрытие всех IN до нулевой позиции → сброс «Разовой проверки» → новый indicator-цикл.

**Ещё открыто:** точный алгоритм расчёта trailing-цены комбо-стопа при добавлении нового IN в группу. Критично ли для движка — решается тестом на snapshots: если `avg_entry` + `unrealized` расходится с snapshots > 5%, тогда важно. [assumption — minor]

## 4. Instop — ДВЕ СЕМАНТИКИ

⚠️ **Важно:** параметр `instop_pct` в API один, но его смысл зависит от режима/стратегии бота. В документации GinArea описаны **два разных поведения**:

### Семантика A — Instop как задержка открытия IN (grid-набор)

**Применяется в:** DEFAULT и DYNAMIC стратегиях (наши живые TEST_1/2/3). [UI docs]

Instop — задержка открытия IN-ордеров, основанная на отклонении цены от локального экстремума в сторону прибыли бота.

#### Базовое правило

IN-ордера набираются **против** направления бота:
- SHORT набирает при росте цены.
- LONG набирает при падении цены.

Instop = задержка разворота цены **в сторону прибыли бота** от экстремума.

#### Сценарий A1 — движение «против» с откатами (нормальный набор)

Пример для SHORT (цена растёт):
- Бот создаёт IN-ордера на расстоянии `grid_step_pct` друг от друга.
- Для каждого следующего уровня ждёт отклонение вниз на `instop_pct` от локального максимума.
- После отклонения — открывает IN.
- Шаг сетки не меняется.

#### Сценарий A2 — первый IN после разворота

Пример для SHORT, цена упала → развернулась → растёт:
- Бот не открывает IN сразу при начале роста.
- Ждёт, пока рост превысит `instop_pct` от минимума.
- Открывает первый IN.

#### Сценарий A3 — безоткатное движение (пропущенные уровни)

Пример для SHORT, цена без откатов летит вверх через N×`grid_step` уровней:
- Бот **не** открывает отдельные IN на каждом уровне.
- Ждёт разворот цены вниз на `instop_pct` от максимума.
- Открывает **ОДИН общий IN**:
  - Размер = N × `order_size` (сумма всех пропущенных уровней). [UI docs]
  - Цена открытия = цена на момент срабатывания instop-триггера (текущая цена). [UI docs]
  - Target = стандартный `target_profit_pct` от цены открытия этого общего IN. [UI docs]

**Этот сценарий — главная причина того, что avg_entry может резко сдвинуться на одном снимке.**

### Семантика B — Instop как расстояние стопа для IN-ордера

**Применяется в:** отдельных режимах работы ботов (каких именно — нужно уточнение).

Цитата из доков: *«Инстоп — определяет расстояние стопа для IN-ордера. Если значение равно 0, параметр не участвует в алгоритме.»*

Пользовательское пояснение: *«этот параметр для начала открытия самого первого ордера»*.

**Интерпретация:** в режимах, где instop работает как «стоп для IN», параметр задаёт расстояние, на котором выставляется защитный стоп от цены открытия IN. То есть — не задержка, а защитный стоп-уровень.

**Открытый вопрос:** в каких именно режимах/стратегиях активна Семантика B? Нужна информация от GinArea или реверс-инжиниринг конфигов других ботов из портфеля (КЛОД_ИМПУЛЬС, BTC_LONG_A/B/C и т.д.).

### Для движка

**Для наших TEST_1/2/3 используется Семантика A** (DEFAULT grid). Алгоритм:

- Каждый минутный бар отслеживает `local_extremum` с момента последнего закрытого IN или старта цикла.
- При достижении next grid level — ставим IN в «ожидание instop».
- Отслеживаем отклонение текущей цены от `extremum`: когда |price − extremum| ≥ `instop_pct` в нужную сторону → открываем IN(ы).
- Если между предыдущим IN и моментом срабатывания instop цена прошла несколько grid-уровней без отката — открываем ОДИН IN размером `N × order_size`.

Для ботов другого типа (при переносе движка) — надо сначала определить, работает ли там Семантика B.

## 5. Indicator (start trigger)

### Параметры Price% (живые TEST_1/2/3)

По скриншоту UI TEST_1: [UI screen]
- Индикатор: **Price%**
- Таймфрейм: **1 min**
- Период: **30**
- Условие входа: **Price% > Значения**
- Значение (%): **0.3**

**Интерпретация:** относительное изменение цены за последние 30 минутных баров, close-to-close, порог > 0.3%. Для SHORT — цена выросла на > 0.3% за 30 мин → можно стартовать. [UI docs + UI screen]

Значение `0.3%` и окно `30 bars 1m` — **не константы**, tunable параметры. В бэктесте — объект grid-search.

### «Разовая проверка» и жизненный цикл indicator

[user] Подтверждено: **«Разовая проверка» = только для первого запуска цикла.**

- **Старт бота** → indicator проверяется → галочка «пройдена» зажигается → бот работает без перепроверки indicator.
- **Out Stop (частичное закрытие)** → indicator НЕ перепроверяется, бот продолжает.
- **Full-close** (вся позиция закрыта: Out Stop закрыл остатки, или Take Profit, или exit indicator) → «Разовая проверка» сбрасывается → новый цикл → ждём новый indicator signal.

**Для движка B1.00:**
- Отслеживать флаг `is_indicator_passed` (сбрасываем в False при старте цикла).
- При `is_indicator_passed == False` → на каждом баре проверяем Price%, если > 0.3% → открываем первый IN (с учётом instop), ставим флаг True.
- При `is_indicator_passed == True` → работаем без indicator, только grid + instop + Out Stop.
- Full-close детектируется как момент, когда `position_size == 0` → сбрасываем флаг.

### Exit indicator

[UI docs] Если выключен — закрытие только по Profit Settings / Take Profit. Если включён — по сигналу, независимо от PnL.

## 6. Boundaries

[UI docs]

**При выходе цены за boundaries:**
- Бот прекращает открывать **новые** IN.
- Существующие IN остаются и могут закрываться за пределами.
- `dsblin=true`: после возврата цены в диапазон — бот не возобновляет автоматически, нужна ручная отмена.

**Симметрия:** работает одинаково на обеих границах (upper и lower) независимо от side. [assumption — логично, подтверждений пока нет]

## 7. PnL и ликвидация

**Биржа:** BitMEX.

### Формулы по типам контрактов

#### Linear USDT-M (BTCUSDT) — используется для SHORT (TEST_1/2/3)

```
Qty: в BTC (order_size = 0.001 BTC)
Notional (USD): qty_BTC × price
Unrealized PnL (USDT):
  SHORT: qty × (entry_price − current_price)
  LONG:  qty × (current_price − entry_price)
Liq_price (SHORT, cross): 
  liq ≈ entry × (1 + (wallet_equity − maintenance_margin) / position_notional)
```

#### Inverse coin-margined (XBTUSD) — используется для LONG (BTC-LONG-C и др.)

```
Qty: в USD контрактах (order_size = $100 = 100 контрактов)
Notional (BTC): contracts / price
Unrealized PnL (BTC):
  LONG:  contracts × (1/entry − 1/current)  → положительно при current > entry
  SHORT: contracts × (1/current − 1/entry)
Liq_price (LONG, cross):
  liq ≈ entry × (1 − IMR_effective)
```

### Для бэктеста — главный подход

**Liq_price в движке = значение, возвращаемое API / трекером.**  
В snapshot'ах поле `liquidation_price` считается платформой с учётом cross-margin pool, maintenance margin, funding, всех ботов одновременно. Воспроизводить эту логику в движке — избыточно.

**Практика:**
1. В валидации B1.00 — сверяем `liq_price` из движка с snapshots. Если расхождение > 5% — разбираемся.
2. В массовых прогонах — используем упрощённую формулу (выше) и **калибруем коэффициент** против snapshot-истории.

### Калибровочные данные из snapshots 24.04

- TEST_1 (linear SHORT): avg_entry ≈ 75,450, liq_price ≈ 101,846 → liq/entry ≈ 1.35.
- Эффективный margin_ratio ~35% → ~2.85× эффективный leverage на cross-пуле.

### Для оператора: асимметрия риска

- **Linear SHORT** при росте цены: unrealized loss растёт ЛИНЕЙНО в USD. Margin requirement тоже.
- **Inverse LONG** при падении цены: unrealized loss в BTC, но BTC в USD ещё и дешевеет → двойной удар в USD-equivalent.
- **Inverse LONG** при росте цены: profit в BTC + BTC дорожает → двойной выигрыш.

Cross-margin → ликвидация по суммарному equity, не per-bot. Защита — контроль `sum(notional)` и `min distance to liq` как метрики портфеля. [уже учтено в STRATEGY_v1 §5]
